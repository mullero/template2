# terraform-mvp

Minimal, production-shaped Terraform for deploying **machote** to Google Cloud
Run with a private Cloud SQL (Postgres) backend, Artifact Registry, Secret
Manager, and keyless GitHub Actions deploys via Workload Identity Federation.

> This is an **MVP skeleton**: one flat root module split into logical files, no
> real business infrastructure beyond the vertical slice. Extend as needed.

## What it creates

| Area        | Resources |
|-------------|-----------|
| Network     | VPC, subnet, Serverless VPC Access connector, private services access (for Cloud SQL private IP) |
| Database    | Cloud SQL Postgres instance (private IP), database, user; password auto-generated into Secret Manager |
| Registry    | Artifact Registry Docker repository |
| Secrets     | `db-password` (managed by TF) + empty secrets for `SECRET_KEY`, Firebase, Neo4j, Gemini (populated out-of-band) |
| Compute     | Two Cloud Run v2 services (backend + frontend) with least-privilege runtime SAs |
| CI identity | Workload Identity Federation pool/provider + deployer SA scoped to your GitHub repo |

## Layout

```
terraform-mvp/
├── versions.tf        providers + GCS backend
├── providers.tf       google / google-beta
├── variables.tf       all inputs
├── main.tf            locals + google_project_service (API enablement)
├── network.tf         VPC / connector / private services access
├── database.tf        Cloud SQL + db + user + generated password
├── registry.tf        Artifact Registry
├── secrets.tf         Secret Manager (db password + external secrets)
├── iam.tf             runtime SAs + Secret access + WIF + deployer SA
├── cloudrun.tf        backend + frontend services + public invoker
├── outputs.tf         URLs, SAs, WIF provider, registry path
├── environments/
│   ├── <env>.tfvars         per-env variables (CHANGE_ME markers)
│   └── <env>.backend.hcl    per-env state prefix + bucket
└── scripts/
    ├── bootstrap-state.sh   create the GCS state bucket (run once)
    ├── enable-apis.sh       pre-enable required APIs
    └── create-secrets.sh    populate external secret values
```

## Prerequisites

- `terraform >= 1.6`, `gcloud` authenticated with sufficient project rights.
- A GCP project per environment (or one project with env-suffixed resources).

## First-time setup (per environment)

```bash
cd terraform-mvp

# 1. Create the remote-state bucket and enable APIs.
./scripts/bootstrap-state.sh <project_id> <bucket_name>
./scripts/enable-apis.sh <project_id>

# 2. Fill in environments/<env>.tfvars and <env>.backend.hcl
#    (replace every CHANGE_ME).

# 3. Init against the env-specific backend.
terraform init -backend-config=environments/prod.backend.hcl

# 4. Plan / apply.
terraform plan  -var-file=environments/prod.tfvars
terraform apply -var-file=environments/prod.tfvars

# 5. Populate the externally-managed secrets.
PROJECT_ID=<project_id> ENV=prod ./scripts/create-secrets.sh
```

## Deploying new images

CI builds images, pushes them to Artifact Registry, then runs
`terraform apply` with `-var="backend_image=..." -var="frontend_image=..."`
(or updates the tfvars). See `.github/workflows/deploy.yml`.

## Security notes

- **No secrets in the repo or state input.** The DB password is generated by
  Terraform; all other secret *values* are added out-of-band via
  `create-secrets.sh` / CI. `CHANGE_ME` in tfvars is a deliberate "must
  override" sentinel.
- Cloud SQL has **no public IP** — access is via the VPC connector + private IP.
- Cloud Run runtime SAs are least-privilege; only the backend SA can read the
  secrets it mounts.
- GitHub Actions authenticates via **WIF** (no downloaded JSON keys); the
  provider is locked to `var.github_repository`.
- `db_deletion_protection` and Postgres PITR backups are on by default for
  staging/prod.

## Caveats

- Neo4j is **not** provisioned here (Cloud Run/Cloud SQL only). Use Neo4j AuraDB
  or a self-managed instance and store `NEO4J_URI` / `NEO4J_PASSWORD` as secrets.
- This module was authored without a local `terraform validate` run in the
  scaffold environment — run `terraform fmt -check` and `terraform validate`
  in CI (see `.github/workflows/terraform.yml`) before first apply.
